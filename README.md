# Смысловой поиск по фото

Поиск по картинкам по смыслу запроса, а не по названию файла или тегам. Пишешь текстом — «я на пляже прошлым летом», «пёс в снегу», «диван как этот, но серый» — и получаешь релевантные фото. Всё считается локально, картинки в облако не уходят.

## Зачем

Когда в архиве десятки тысяч снимков, нужное вручную не найдёшь. Поиск по имени файла и тегам не работает, потому что не знает, что изображено на картинке. Плюс языковой барьер: запрос на русском, а описания в датасетах на английском.

Целимся в две аудитории. Обычный пользователь с личным архивом — ему нужна одна строка поиска и сетка результатов, без настроек (версия на `/`). И бизнес (маркетплейсы, фотостоки, медиа-архивы, модерация) — там нужен движок для каталога, выбор модели, точный режим и API (версия на `/business`).

## Как устроено

Каждая картинка и текстовый запрос переводятся в вектор (эмбеддинг) моделями CLIP и BLIP, дальше поиск идёт по векторам через FAISS. Русский запрос переводится на английский локально через Opus-MT, так что язык ввода не важен. Тройной ансамбль с BLIP-2 даёт прибавку к качеству, точный режим (ITM-реранж) уточняет топ выдачи.

Инференс на CPU, без видеокарты. Recall@1 ≈ 61%, поиск < 1 сек на CPU.

## Что в репозитории

- `app/` — сам сервис: `engine.py` (поиск), `main.py` (веб-API на FastAPI), `static/` (две версии интерфейса).
- `Final/` — Docker-обёртка: `Dockerfile`, `docker-compose.yml`, `requirements-web.txt`, `run_demo.sh`.

## Чего в репозитории НЕТ (нужно положить рядом самому)

Модели, данные и эмбеддинги большие, поэтому в git их не храним. Перед запуском положи в корень проекта три папки:

- `MODELS/` — веса моделей (папки с HuggingFace):
  - `clip-vit-large-patch14`
  - `blip-itm-large-flickr`
  - `opus-mt-ru-en` (перевод ru→en)
  - `blip-2` (тройной ансамбль, грузится по необходимости)
  - `blip-caption` (подписи к загруженным фото)
- `DATA/` — картинки и подписи:
  - `captions.csv` (колонки `image_id, file_name, caption_en, caption_ru`)
  - `val2017/` — картинки
  - `coco_balanced_15k/images/` — картинки
- `embeddings_output_20k/` — заранее посчитанные эмбеддинги:
  - `image_ids_order.csv`, `caption_ids_order.csv`
  - `clip_image_embeddings.npy`, `blip_image_embeddings.npy`
  - `clip_text_embeddings.npy`, `blip_text_embeddings.npy`
  - `image_embeds_BLIP-2.npy`

Без этих трёх папок сервис не поднимется — движок читает их при старте.

## Запуск

Через Docker (проще всего). Из папки `Final/`:

```
docker compose up --build
```

Модели, данные и эмбеддинги подключаются томами с хоста (в образ не кладутся), поэтому три папки выше должны лежать в корне проекта. Сайт поднимется на порту `8000`:

- `http://localhost:8000/` — версия для человека (B2C)
- `http://localhost:8000/business` — версия для бизнеса (B2B)

## Демо в общей сети (например, для жюри)

Скрипт `Final/run_demo.sh` печатает адрес ноутбука в сети и поднимает сервис:

```
bash Final/run_demo.sh
```

Все, кто в том же Wi-Fi, открывают `http://<адрес-ноутбука>:8000/`. Порт `8000` должен быть свободен (останови dev-сервер, если он запущен) и не закрыт фаерволом. Если раздаёшь Wi-Fi с телефона — проверь заранее, что в точке доступа не включена «изоляция клиентов», иначе устройства не увидят друг друга.
